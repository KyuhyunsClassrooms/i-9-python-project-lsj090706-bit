# 최종 보고서

## 1. 프로그램 주제와 목적

내 프로그램의 주제:나의 예산과 맛 취향에 따른 맞춤형 디저트 추천 및 가상 주문 시뮬레이터

이 프로그램을 만든 이유:학교 매점이나 카페에서 학생들이 한정된 용돈(예산) 안에서 간식을 고를 때 선택 장애를 겪는 경우가 많다. 자신이 가진 금액 조건과 원하는 맛(달콤함, 상큼함, 짭짤함 등)을 입력하면 실시간으로 최적의 메뉴를 필터링해 추천해 주고, 가상 주문을 통해 결제 및 잔액 계산까지 도와주는 프로그램을 만들어 합리적인 소비 선택을 지원하고자 제작하였다.


---

## 2. 문제 분해 및 모델링

### 입력
사용자가 현재 가지고 있는 예산 (정수 데이터, 예: 5000)

사용자가 원하는 디저트 맛 취향 (문자열 데이터, 예: "달콤함", "상큼함", "짭짤함")

추천된 리스트 중 구매하려는 메뉴의 인덱스 번호 및 구매 수량

### 처리
필터링: 전체 디저트 데이터가 담긴 2차원 리스트를 순회하며 가격 <= 예산 이고 맛 == 원하는 맛인 항목만 골라낸다.

예외 제어: 조건에 맞는 메뉴가 없거나 사용자가 잘못된 메뉴 번호를 입력했을 때의 처리를 수행한다.

연산: 선택한 메뉴의 가격 * 수량으로 총 주문 금액을 계산하고, 현재 예산과 비교하여 결제 가능 여부를 판별한다.


### 출력
조건에 부합하는 개인 맞춤형 디저트 추천 메뉴판 리스트

가상 주문 결과 영수증 (선택 메뉴 이름, 수량, 총 주문 금액, 구매 성공 여부, 결제 후 남은 잔액)

## 3. 사용한 2차원 리스트 설명

2차원 리스트 이름:desserts

한 행의 의미:추천 시스템에 등록된 디저트 1종류의 통합 상세 정보

각 열의 의미:열 번호,의미,실제 예시
0번 열,디저트 이름 (문자열),"""초코 마카롱"""
1번 열,판매 가격 (정수형),2500
2번 열,맛 분류 (문자열),"""달콤함"""
3번 열,상품 특징 설명 (문자열),"""인기 최고 디저트"""



이 2차원 리스트를 사용한 이유:디저트라는 하나의 객체는 이름, 가격, 맛, 특징 등 다양한 형태의 이종 데이터를 동시에 가집니다. 단일 리스트 여러 개를 쓰면 인덱스를 동기화하기 어렵지만, 2차원 리스트를 사용하면 '하나의 행'에 특정 디저트의 모든 정보를 묶어서 체계적으로 관리하고 관리·확장하기 용이하기 때문입니다.

## 5. 조건문과 반복문 활용

### 조건문 사용 부분

어떤 조건을 검사했는가?
1) 디저트를 추천할 때: category == taste and price <= budget (맛이 같고 가격이 예산 이하인가?)
2) 메뉴를 선택할 때: choice in valid_ids (추천된 메뉴판에 존재하는 번호를 입력했는가?)
3) 결제할 때: total_price > budget (총 주문 금액이 보유한 예산을 초과하는가?)


### 반복문 사용 부분

무엇을 반복 처리했는가?
1) for i in range(len(data))를 통해 desserts 리스트의 0번 행부터 마지막 행까지 차례대로 접근하여 데이터를 순회 검사했습니다.
2) while True 무한 루프를 사용하여 사용자가 메뉴 번호를 잘못 입력했을 때, 올바른 추천 번호를 입력할 때까지 계속해서 재입력을 요구하도록 반복 처리했습니다.


### 조건문과 반복문이 함께 작동한 부분
find_recommendations 함수 내부에서 반복문(for)이 2차원 리스트의 각 행을 하나씩 짚을 때마다, 그 내부에 위치한 조건문(if)이 해당 행의 가격과 맛 데이터를 검사합니다. 즉, 반복문이 전체 데이터를 훑는 과정 속에서 조건문이 필터 역할을 수행하며 매칭되는 데이터만 선별해 내는 구조로 함께 작동합니다.

## 7. 테스트와 오류 수정

### 테스트 1: 정상 사례

입력:예산 4000 / 맛 짭짤함 / 메뉴 번호 5 / 수량 1

예상 결과:가격이 3,000원인 '치즈 나초'와 2,800원인 '소금빵'이 추천됨 -> 5번 소금빵 1개 선택 시 2,800원이 결제되고 잔액 1,200원 출력됨.

실제 결과:예상 결과와 일치하게 정상 작동함.


### 테스트 2: 예외 또는 경계 사례

입력:예산 1000 (매우 적은 금액 입력) / 맛 상큼함

예상 결과:1,000원 이하인 상큼한 디저트가 리스트에 없으므로, "조건에 맞는 디저트가 매점에 없습니다..."라는 예외 안내 문구가 출력되어야 함.

실제 결과:valid_menu_ids가 빈 리스트가 되면서 main()의 예외 처리 분기문이 작동하여 정상적으로 안내 메시지를 출력함.


### 발견한 오류와 수정한 방법
오류 상황: 처음에 메뉴 번호를 고를 때 추천 메뉴판에 없는 번호(예: 추천은 0번만 나왔는데 2번 입력)를 입력해도 프로그램이 그대로 진행되어 잘못된 영수증이 출력되거나 인덱스 범위 초과 에러(IndexError)가 발생함.

수정 방법: process_order 함수 내부에 while True 반복문과 if choice in valid_ids: 조건문을 추가하여, 사용자가 올바른 추천 번호를 입력했을 때만 break로 루프를 탈출하고 그렇지 않으면 에러 메시지와 함께 다시 입력받도록 보완하여 결함성을 해결함.

## 8. Gemini 활용 내용 요약

Gemini에게 도움받은 부분:수행평가 조건(2차원 리스트, 함수 분리 등)에 딱 들어맞으면서 일상적이고 흥미로운 '디저트 추천 및 가상 주문'이라는 대주제를 선정하는 데 도움을 받았습니다. 프로그램의 구조를 기능별로 쪼개어 함수 4~5개로 분리하는 모듈화 설계안과 다중 조건문 예시 코드를 제공받아 개발 시간을 단축했습니다.

Gemini가 제안했지만 내가 수정하거나 사용하지 않은 부분:Gemini는 처음에 추천된 ID를 검증하는 로직을 main 함수 쪽에 다소 복잡하게 섞어서 제안했으나, 코드가 한눈에 들어오지 않아 이를 직관적으로 이해할 수 있도록 process_order라는 별도의 주문 처리 함수 안으로 검증 코드를 묶어서 재배치하였습니다.



최종 코드에서 내가 직접 판단하거나 수정한 부분:사용자 경험을 높이기 위해 출력 화면의 가독성을 다듬었습니다. 영수증이 출력되는 부분에 점선("-" * 30) 텍스트 데코레이션을 추가하여 실제 영수증처럼 보이게 수정하였고, 변수명들을 my_budget, selected_name 등 직관적인 한글 의미에 매칭되도록 가독성을 높여 직접 수정했습니다.

## 9. 한계와 개선 방안

현재 프로그램의 한계:현재 프로그램은 사용자가 예산이나 수량을 입력할 때 실수로 숫자가 아닌 '문자(예: 오천원, 두개)'를 입력하면 파이썬 자체에서 ValueError가 발생하며 프로그램이 강제 종료되는 한계가 있습니다. 또한, 한 번에 한 가지 메뉴만 골라 주문할 수 있어 여러 개의 디저트를 동시에 장바구니에 담아 주문하는 기능이 부족합니다.


다시 개선한다면 추가하고 싶은 기능:try-except 구문을 배워서 문자열 입력 에러를 안전하게 방어하는 예외 처리 기능을 추가하고 싶습니다. 또한, 반복문을 확장하여 사용자가 "더 주문하시겠습니까?"라는 질문에 "종료"라고 답할 때까지 여러 디저트를 장바구니 리스트에 계속 누적하여 최종 합산 금액을 계산해 주는 '장바구니 기능'을 추가해 보고 싶습니다.

## 10. 코드인터뷰 준비

다음 질문에 답할 수 있도록 준비하세요.

1. 내 프로그램의 입력-처리-출력은 무엇인가?사용자의 예산과 맛 취향을 입력받아, 2차원 리스트를 탐색하여 조건에 맞는 디저트를 필터링하고 금액을 계산하는 처리를 거쳐, 최종 추천 메뉴판과 가상 주문 영수증을 출력합니다.
2. 2차원 리스트에서 한 행과 각 열은 무엇을 의미하는가?한 행은 디저트 1종류의 통합 정보를 의미하며, 각 열은 순서대로 0번: 이름, 1번: 가격, 2번: 맛 분류, 3번: 특징 설명을 의미합니다.
3. 가장 중요한 함수는 무엇이고 어떤 역할을 하는가?find_recommendations 함수입니다. 전체 디저트 데이터에서 사용자의 입력 조건(맛, 가격)에 일치하는 데이터만 for문과 if문으로 솎아내어 맞춤형 메뉴판을 구성하는 핵심 알고리즘 역할을 합니다.
4. 조건문과 반복문은 어디에서 사용되었는가?반복문은 2차원 리스트를 순회할 때(for), 그리고 올바른 번호를 재입력받을 때(while) 사용되었습니다. 조건문은 맛과 예산 조건을 필터링할 때(if category == taste), 그리고 결제 시 잔액을 비교할 때(if total_price > budget) 사용되었습니다.
5. Gemini가 도와준 부분과 내가 직접 수정한 부분은 무엇인가?Gemini는 전체적인 디저트 시뮬레이터 주제 기획과 함수 분리 설계의 뼈대를 제공해 주었습니다. 저는 코드의 가독성을 위해 영수증 UI 디자인을 개선하고, 입력값 검증 코드를 process_order 함수로 묶어 독립시키는 구조적 수정을 직접 판단하여 수행했습니다.
